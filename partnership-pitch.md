# EatDiscounted Partner Outreach

## Writing-Review Diagnosis

This is the direction to preserve:

- First person, because this is strongest when it sounds like Shari built something from a real user problem.
- Specific, not inflated. No fake traction, no generic "partnership opportunity" language.
- Clear about the ask: a lightweight read-only restaurant feed/API.
- Clear about the boundary: no user data, payment data, personalized pricing, or redemption access.
- Respectful of each platform. EatDiscounted should sound like a discovery layer that sends users back, not a replacement.
- Close on partner upside: more app opens, better recall at the spending moment, cleaner referrals, and no risk to their user relationship.

The edit below removes the more generic business-development phrasing and makes each note more concrete to the platform.

## One-Pager Copy

### EatDiscounted: one restaurant search, the right rewards app

EatDiscounted is a free NYC restaurant search tool that helps diners answer one question:

> I know where I want to eat. Which app should I check?

Users do not always start by browsing reward apps. A lot of the time, they start with a restaurant. EatDiscounted checks the available dining rewards and discount programs, shows which ones are relevant, and sends the user back to the platform that handles the actual offer, booking, purchase, or app experience.

That is why partner data matters. Without it, EatDiscounted can only make partial guesses from public pages, which means participating platforms may be missing at the exact moment a user is ready to act. With a basic feed, the product can answer the user honestly and send them to the right app at the right time.

### How it works

1. A user searches a restaurant by name.
2. EatDiscounted checks available dining reward and discount programs.
3. Results show which platforms may have cashback, credits, points, or discounts.
4. The user clicks through to the platform that handles the next step.

### What a partner feed needs

The first version can be intentionally small:

- Restaurant name
- Address or coordinates
- City/neighborhood
- Offer type or reward type
- Public listing URL or app deep link, if available
- Last-updated timestamp, if available

EatDiscounted does not need user-level data, payment data, redemption data, or personalized offer logic for a first pilot.

### Why it helps partners

| Partner benefit | Why it matters |
| --- | --- |
| Users remember to check you | EatDiscounted shows the platform at the restaurant decision moment. |
| Traffic goes back to your app | The offer, booking, purchase, and account experience stay with you. |
| Better accuracy | A partner feed avoids stale or inaccurate public-search guesses. |
| Low integration lift | A read-only feed is enough for a useful pilot. |
| Free user-facing discovery | Diners can find you without another ad placement or paid listing. |
| Better timing | Users see the platform when they are already close to spending money. |
| Cleaner acquisition | Users arrive because they searched a restaurant you can actually serve. |

### Why it helps diners

Diners no longer have to open several apps one by one before choosing where to book, buy, or claim a reward. They can search once, compare options, and go directly to the platform that makes sense for that restaurant.
